# Ai agent 实现和应用

## 1. 实现原理

agent的实现，很大一部分是基于大模型提供的function_call和tool_calls能力，

### 1.1 ``function_call`` 是调用外部函数的一种机制，主要能力包括：
1. 扩展 AI 能力
- 语言模型本身只能生成文本，无法直接访问外部数据或执行复杂计算。
- 通过 function_call，AI 可以调用预定义的函数，从而获取实时数据、执行计算或操作外部系统。

2. 结构化输出
- function_call 要求 AI 生成结构化的请求（如 JSON 格式），明确指定需要调用的函数及其参数。 这种结构化输出便于后续系统解析和执行。
3. 动态任务处理
- 根据用户请求的内容，AI 动态决定是否需要调用函数，以及调用哪个函数。例如，用户询问天气时，AI 会调用天气查询函数；用户需要计算时，AI 会调用计算函数。

#### 1.1.1 示例:
1. 输入 ``北京天气如何``
2. 解析识别出天气查询函数
3. 生成function_call调用返回
   ```json
    {
        "function_call": {
            "name": "get_weather",
            "arguments": {
                "city": "北京"
            }
        }
    }
   ```
4. 调用函数，返回结果
   ```json
    {
        "weather": "晴",
        "temperature": "25℃"
    }
   ```
5. AI根据结果生成回复


### 1.2 ``tool_calls``是 function_call 的扩展，支持同时调用多个工具或函数。它的作用包括：

1. 多工具协作
- 一个任务可能需要多个工具协作完成。例如，用户请求“查询纽约的天气并翻译成中文”，AI 需要调用天气查询工具和翻译工具。
- tool_calls 允许 AI 同时生成多个函数调用请求。

2. 复杂任务分解
- 对于复杂任务，AI 可以将其分解为多个子任务，分别调用不同的工具完成。 例如，用户请求“帮我订一张从北京到上海的机票，并设置一个出发提醒”，AI 可以分别调用订票工具和提醒工具。

3. 提高效率
- 通过一次性生成多个工具调用请求，减少与用户的交互次数，提高任务处理效率

#### 1.2.1 示例
1. 输入：“查询纽约的天气并翻译成中文。
2. 请求解析，识别出查询天气、翻译中文两个函数调用
3. 返回解析结果
   ```json
   {
    "tool_calls": [
                {
                    "id": "tool_1",
                    "type": "function",
                    "function": {
                        "name": "get_weather",
                        "arguments": {
                            "city": "纽约"
                        }
                    }
                },
                {
                    "id": "tool_2",
                    "type": "function",
                    "function": {
                        "name": "translate",
                        "arguments": {
                            "text": "The weather in New York is sunny with a temperature of 22°C.",
                            "target_language": "中文"
                        }
                    }
                }
        ]
   }
   ```
4. 依次调用函数实现
5. 结果返回
   ```json
    {
       "weather": "晴",
       "temperature": "22℃"
    }

    {
        "output": "纽约的天气晴朗，气温为22摄氏度。"
    }
   ```
6. AI根据结果生成回复,整合为自然语言

#### 1.3 ``mcp`` 协议支持

mcp和 function call 协议基本一致，都是通过llm解析用户输入，生成调用请求的参数，调用外部函数，返回结果，生成回复。

mcp是否需要显示声明调用协议，取决于模型是否支持mcp协议，如果支持，可以不显示声明，直接调用函数，否则需要显示声明。例如：阿里云的qianwen模型，默认支持mcp协议，可以不显示声明，直接调用函数，否则需要显示声明。

但是通常建议tool_call和mcp协议(即在systemPrompt中显示声明模型对于方法的调用)同时支持，这样可以兼容更多的模型，同时也可以兼容更多的场景。


#### 1.4 function call 和 mcp 对比


| 特性	    |   Function Calling (OpenAI 标准)	   |     MCP (自定义/阿里云协议) |
| -- | --| -- |
| 协议来源	  | OpenAI 定义的通用工具调用规范	         |  阿里云/自定义的模型上下文协议 |
| 标准化程度    |	行业通用标准（如GPT-4o、Claude等均兼容）|	通常为特定厂商或场景定制（如千问、灵积模型） |
| 工具描述格式 |	严格遵循OpenAI的JSON Schema	     |         可能使用自定义的DSL或简化格式 |
| 多工具协作	| 支持单次请求调用多个工具	              |  通常需显式串联多个操作 |
| 触发方式	 |    模型自动识别工具并调用 |	需在Prompt中显式声明协议格式 |
| 工具描述	 |    强制要求参数类型、描述等元信息	| 通常更自由，但依赖厂商实现 |
| 错误处理	 |    标准化错误码（如tool_not_found）|	自定义错误响应（如阿里云的InvalidParameter） |
| 适用模型	 |    OpenAI、Anthropic等通用模型 |	阿里云千问、DashScope等特定平台 |
| 开发复杂度 | 低（LangChain4j自动封装）|	中（需适配厂商SDK） |
